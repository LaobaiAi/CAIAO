# CAIAO 蒸馏方法论

> 从已有复杂系统中提取领域能力，封装为 CAIAO 标准 Server 的完整方法论。
> 基于从已有大型工程系统中蒸馏领域能力的实践经验。

## 什么是蒸馏

从已有的复杂系统中，提取某一领域能力的纯计算逻辑，去除多用户、数据库、网络、前端等运行时依赖，封装为一个独立、契约化、LLM 可直接调用的原子 Server。

| 蒸馏前（源系统） | 蒸馏后（CAIAO Server） |
|:---|:---|
| 单体/微服务架构，依赖链路深 | 单一 `.py` 文件，零间接依赖 |
| 通过 HTTP/gRPC 通信 | 通过 `call_tool(name, input)` 进程内/stdio 通信 |
| 耦合业务逻辑与基础设施 | 纯计算逻辑，输入 JSON→输出 JSON |
| 面向人类开发者调试 | AI-ready：工具描述 + JSON Schema 完备声明 |

## 六步蒸馏流程

```
源系统分析 → 领域边界划分 → 数据模型抽象 → 独立实现 → 契约验证 → 编排演示
```

### Step 1: 源系统分析

深入研读源系统的能力定义文件、核心计算模块、数据模型和调用链路。

实际蒸馏经验：分析源系统的多个技能包（类型定义、截面库、荷载与边界、分析引擎、规范校核、报告生成），提取领域特有逻辑，忽略用户管理、权限等基础设施。

### Step 2: 领域边界划分

将全流程拆解为 3-6 个原子能力：

拆分原则：
- 单一职责——一个 Server 只做一件事
- 无状态——输入足够，不依赖外部隐式状态
- 可独立验证——给定输入，输出可单独测试
- 自然边界——沿领域知识边界切分

错误示范：一个 Server 既建模又分析又出报告。
正确示范：Generator → Loader → Runner → Checker → Reporter。

### Step 3: 数据模型抽象

每个 Server 的输入/输出定义为 JSON Schema：
- 自描述——每个字段有 description，AI 可无歧义理解
- 最小完备——只包含必要字段
- 类型严格——物理量用 number，索引用 integer，ID 用 string
- 可扩展——不设 `additionalProperties: false`

### Step 4: 独立实现

关键指导原则：
- 优先纯 Python——依赖项只保留 numpy 等基础库
- 降级方案内置——可选依赖不可用时自动 fallback
- 内置数据——截面库、材料库直接硬编码
- 简化为先——先用简化算法覆盖主要场景，预留扩展接口

### Step 5: 契约验证

使用 `jsonschema` 库对每个 Server 的输入输出做 Schema 校验。

### Step 6: 编排演示

新建 Pipeline Server（合并 Server），顺序调用原子 Server，通过 CLI 入口提供一键运行。

## 关键设计决策速查表

> 以下为从参考实现中提炼的通用决策框架，具体选项和推荐取决于项目上下文。

| 决策点 | 选项 A | 选项 B | 推荐 | 理由 |
|:---|:---|:---|:---|:---|
| 计算引擎 | 外部专业求解器 | 自研简化算法 | 视场景而定 | 权衡精度与安装门槛 |
| 数据处理 | 精确完整处理 | 简化等效法 | 视场景而定 | 演示精度通常足够 |
| 规则校验 | 完整实现全部条款 | 核心公式 + 预留扩展 | B | 覆盖主要场景 |
| Server 通信 | subprocess + JSON 管道 | 进程内直接调用 | B（早期阶段） | 开发效率高 |
| 输出生成 | 纯模板引擎 | 模板 + 字符串 fallback | B | 环境受限时仍可工作 |
| 内置数据 | 外部文件加载 | 硬编码内置数据 | B | 无文件依赖 |

## 常见陷阱

| 陷阱 | 症状 | 对策 |
|:---|:---|:---|
| 依赖残留 | Server A import Server B 的类 | 纯逻辑提取为独立工具函数 |
| Schema 漂移 | Pipeline 串联时下游解析失败 | 每个 Server 输出后做 jsonschema.validate() |
| 硬编码路径 | 输出文件写到绝对路径 | 输出目录通过参数传入 |
| AI 描述不足 | LLM 不知道何时调用工具 | @tool 的 description 写清适用场景 |
| 过度简化 | 算法太简陋 | 核心计算参考教材标准实现 |
| 边界遗漏 | 零跨、单层等情况崩溃 | 实现时考虑边界：最小/最大尺寸、空数组 |

## 蒸馏效率参考

| 阶段 | 参考工时 | 关键耗时 |
|:---|:---:|:---|
| 源系统分析 | 1-2h | 理解数据流和算法粒度 |
| Schema 设计 | 1-2h | 与源系统数据结构对齐 |
| 原子 Server 实现（每 Server） | 1-3h | 算法翻译 + 边界处理 |
| Pipeline 编排 | 0.5-1h | 数据串联和错误传递 |
| CLI + 输出 | 1h | 美化输出、文件组织 |
| 测试 + 文档 | 1-2h | 端到端验证 |
| **合计** | **10-20h** | 3-5 个原子 Server 规模 |
