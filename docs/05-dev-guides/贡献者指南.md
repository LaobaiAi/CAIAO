# CAIAO 项目贡献者指南

> 基于钢框架设计项目的 CONTRIBUTING.md 提炼。
> 适用于所有 CAIAO 生态项目的开发规范和协作流程。

---

## 一、分支策略

| 分支类型 | 模式 | 用途 |
|----------|------|------|
| master | — | 生产就绪，始终可部署 |
| Feature | `feature/` | 新能力 |
| Fix | `fix/` | Bug 修复 |
| Refactor | `refactor/` | 代码重构，无行为变更 |

规则：
- 从 master 分支，合并回 master
- 保持分支短命（理想 3 天以内）
- 开 PR 前 rebase 到 master
- 绝不 force-push 到 master

## 二、Commit 规范（Conventional Commits 1.0.0）

### 格式

```
<type>(<scope>): <description>

[optional body]

[optional footer]
```

### Types

| Type | 用途 |
|------|------|
| `feat` | 新功能或能力 |
| `fix` | Bug 修复 |
| `refactor` | 代码重构无行为变更 |
| `style` | 格式、空格（无逻辑变更） |
| `docs` | 仅文档 |
| `test` | 添加或更新测试 |
| `chore` | 工具、依赖、构建脚本 |
| `perf` | 性能改进 |

### Scopes（项目特定）

| Scope | 适用范围 |
|-------|---------|
| `frame` | 框架生成 Server |
| `load` | 荷载生成 Server |
| `fea` | FEA 运行器 Server |
| `check` | 规范校核 Server |
| `report` | 报告生成 Server |
| `pipeline` | 管线编排器 |
| `cli` | CLI 入口 |
| `schema` | JSON Schema 定义 |
| `caiao` | CAIAO 基类、Hub 或协议 |
| `llm` | LLM 网关、Agent、参数提取器 |
| `api` | Web API Server |
| `frontend` | 前端应用 |
| `review` | 代码审查报告 |
| `docs` | 文档 |
| `ci` | CI/CD |

### 规则

1. Type 和 scope 是必须的
2. Description 用英文，祈使语气，小写，不超过 72 字符
3. 一个 commit 一个逻辑变更
4. Breaking changes：在 type/scope 后加 `!`，footer 加 `BREAKING CHANGE:`

### 禁止

- `update`、`changes`、`wip`、`tmp`、`misc` 作为 type
- 模糊描述：`fix bug`、`update code`
- 修改已发布的 commit
- 混合无关变更到一个 commit
- 空 commit message

## 三、代码风格

### Python

- Black 格式化，行长度 120
- 所有函数签名加类型标注
- Google 风格 docstring（公开函数）
- 导入顺序：标准库 → 第三方 → 本地，组内字母排序
- 不用 `*` 导入
- 用 logging 而非 print()

### CAIAO Server 约定

- 始终返回结构化 dict，绝不返回原始字符串
- 用 json.dumps 序列化结果
- 捕获所有异常返回 `{"error": str(e)}`
- 工具名用 snake_case
- 每台 Server 原子化——单一职责，零跨域逻辑
- Pipeline Server 仅编排——绝不嵌入领域逻辑

## 四、Pull Request 流程

### 开 PR 前

- 分支已 rebase 到最新 master
- 所有测试本地通过
- 手动 smoke test 通过
- Commit 历史清晰

### PR 标题

与 commit 相同格式：`<type>(<scope>): <description>`

### PR 内容

- Summary：做了什么和为什么
- Test plan：测试步骤
- Screenshots（如报告模板变更）

### 审查要求

- 至少一个 approving review
- 所有 CI 检查绿色
- 无未解决的 review 线程
- 审查者验证完整管线

---

## 五、数据流约定

每个 Server 从前一步读取 JSON，写入 JSON 给下一步。

规则：
- 每个 Server 间边界必须符合 schemas/ 中对应的 JSON Schema
- 绝不破坏向后兼容性（除非 BREAKING CHANGE commit）
- 可选字段必须有合理默认值
