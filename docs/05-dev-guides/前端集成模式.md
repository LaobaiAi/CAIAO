# CAIAO 前端集成模式

> CAIAO Server 如何被前端消费——两种项目采取了不同的路径，各有适用场景。

## 模式 A：MCP SDK 实现的前端集成

> 以下为来自参考实现的案例示例。核心集成思路是通过 Gateway + WebSocket + REST 的组合连接前端与 CAIAO Server 生态，具体技术选型（框架、3D 引擎、通信协议）可根据项目需求替换。

### 架构（参考实现）
```
前端（Next.js + TypeScript）
  │
  ├─ WebSocket ──→ Gateway AgentLoop
  │                  │
  │                  └─→ CAIAOClientHub → CAIAO Server 子进程
  │
  ├─ REST API ──→ Gateway (FastAPI)
  │                │
  │                ├─ /tools   → hub.list_tools()
  │                ├─ /servers → hub.get_all_status()
  │                └─ /health  → hub.get_all_health()
  │
  └─ WebRTC ──→ Unity (3D visualization)
```

### 关键特性
- **Server 管理仪表盘**：前端通过 manager_server 的 24 个工具管理整个 Server 生态
- **工具按需发现**：LLM Agent 通过 Hub 动态发现工具，前端不硬编码工具列表
- **多面板布局**：结构模型、应力云图、倒塌动画、Unity 3D 视图分布在独立面板中
- **WebSocket 状态同步**：分析进度、倒塌步骤通过 WebSocket 推送到前端

### 前端组件对应 Server
| 前端组件 | 消费的 Server / 工具 |
|----------|---------------------|
| frame-visualization.tsx | frame_generator → generate_frame |
| verification-panel.tsx | opensees_server → high_fidelity_analysis |
| unity-video-panel.tsx | unity_simulator → 所有 4 个工具 |
| floating-toolbar.tsx | quick_analysis_server → quick_analysis |
| server-manager.tsx | manager_server → 所有 24 个工具 |

## 模式 B：轻量实现的前端集成

> 以下为来自参考实现的案例示例。核心集成思路是通过单进程 FastAPI 服务统一托管后端 API 和前端静态文件，适合独立工具型项目。

### 架构（参考实现）
```
前端（React 18 + Vite + TypeScript）
  │
  ├─ REST API ──→ Web API Server (FastAPI, 合并 Server)
  │                │
  │                └─→ Hub → CAIAO Server（进程内直调）
  │
  └─ Static files ── 由同一个 FastAPI 进程托管
```

### 关键特性
- **单进程服务**：后端 API + 前端静态文件由同一个 FastAPI 进程提供，部署极简
- **3D 可视化**：Three.js（`@react-three/fiber`）渲染钢框架模型
- **Zustand 状态管理**：前端状态集中在 `useStore.ts`
- **SSE 流式**：LLM 响应通过 Server-Sent Events 流式推送到前端
- **项目持久化**：API 自动归档运行结果

### 前端 API 端点
| 端点 | 方法 | 用途 |
|------|------|------|
| `/api/health` | GET | 健康检查 |
| `/api/run_pipeline` | POST | 工程模式全流程 |
| `/api/llm_param` | POST | LLM 参数提取 |
| `/api/llm_agent` | POST | LLM Agent 全自动 |
| `/api/llm/stream` | POST | SSE 流式对话 |
| `/api/sections` | GET | 截面/材料库（前端下拉框动态获取） |
| `/api/colormap` | GET | 应力比颜色映射（前后端共用定义） |
| `/api/tools` | GET | 所有注册工具列表 |
| `/api/project/save` | POST | 手动保存项目 |
| `/api/project/list` | GET | 列出已保存项目 |
| `/api/project/load` | POST | 加载已保存项目 |

### 前后端共享数据
`defaults.py` 中的 `COLORMAP_STOPS` 和 `COLORMAP_THRESHOLDS` 通过 `/api/colormap` 和 `/api/sections` 端点提供给前端，确保颜色渲染和下拉选项与后端保持一致。

## 两种前端集成模式的对比

| 维度 | MCP SDK 实现 | 轻量实现 |
|------|-------------|---------|
| 部署复杂度 | 高（Gateway + 多个子进程 + Unity + 前端 dev server） | 低（单个 FastAPI 进程 + 前端静态文件） |
| 实时通信 | WebSocket（双向） | SSE（单向流式）+ REST |
| 3D 引擎 | Unity WebRTC（专业游戏引擎） | Three.js（Web 标准） |
| Server 管理 UI | 完整仪表盘（22 台 Server 状态） | 无需（12 台 Server 进程内） |
| 适用场景 | 多求解器平台，需进程级隔离 | 独立工具，单进程即可 |

## 前端开发注意事项

1. **不要硬编码工具名**：工具列表应从 `/api/tools` 或 `hub.list_all_tools()` 动态获取
2. **颜色映射统一来源**：前后端必须使用同一份颜色定义（`defaults.py` → API → 前端）
3. **大结果数据裁剪**：管线中间结果可能很大（如完整分析结果 >100KB），前端传输前应裁剪——只传摘要和可视化需要的数据
4. **状态恢复**：前端应能从对话历史中恢复完整状态（结构模型 + 分析结果 + 动画），不仅是文本消息
5. **错误显示**：Server 返回的 `{"error": "..."}` 应在前端以人类可读的方式呈现，而非 dump 原始 JSON
6. **多语言就绪**：所有用户可见文字应通过 `t(key, lang)` 函数获取，不在 JSX 中硬编码
