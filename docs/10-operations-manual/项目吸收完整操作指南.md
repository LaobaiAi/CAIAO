# 项目吸收完整操作指南

> 基于两个真实案例——蒸汽轮机厂房（高难度）和 Abaqus_Collapse（低难度）——
> 提炼的完整吸收方法论。每次吸收新项目时按此流程执行。

---

## 一、核心认知

**吸收不是改造——是注册**。理想的吸收只需提供两个标准数据文件（几何清单 + 拆除序列），不改任何 Server 代码。如果吸收过程需要改架构，说明原项目的开发方式偏离了主项目约定。

吸收成本公式：**兼容维度越多 → 成本越低**。其中最关键的维度是「架构思想」（是否 CAIAO 化）和「工具契约」（输入输出格式）。

---

## 二、吸收六步骤

### 步骤 1：兼容性评估（六维度表格）

拿到待吸收项目后，立即制作此表：

| 维度 | 检查内容 | 兼容？ |
|------|---------|--------|
| **架构思想** | 是否遵循每能力一台 Server？是否使用 Hub 路由？ | |
| **工具契约** | 输入输出是否 JSON Schema 化？错误格式是否统一？ | |
| **构件标识** | 构件是否有 element_type、floor、importance 等元数据？ | |
| **拆除/编排逻辑** | 是声明式（数据驱动）还是命令式（硬编码步骤）？ | |
| **配置格式** | 配置文件是否符合主项目 schema？ | |
| **执行路径** | 是否通过 CAIAO Hub 执行？是否走标准子进程/spawn 模式？ | |
| **几何 API** | 是否使用 _common 低级 API？是否避免 bpy.ops 操作符？ | |
| **Python 环境** | 是否需要隔离解释器（Abaqus/Blender 内嵌 Python）？ | |

评估规则：
- 兼容 6+ 维度 → 吸收成本 <2 小时
- 兼容 4-5 维度 → 吸收成本 2-4 小时
- 兼容 2-3 维度 → 吸收成本 4-8 小时
- 兼容 0-1 维度 → 吸收成本 >8 小时，考虑是否值得

**架构思想和工具契约是加权最高的两个维度**——如果这两项兼容，其他不兼容只是格式转换。如果这两项不兼容，本质上是架构重设计。

### 步骤 2：提取几何定义

**目标**：将原项目的几何生成逻辑转为标准数据文件。不保留原始生成代码，只保留数据。

**方法**：
1. 运行一次原项目的生成脚本
2. 导出所有构件的数据——唯一 ID、类型、位置、尺寸、旋转、材质、元数据标签
3. 存为标准格式（JSON/YAML）
4. 原代码可以删除

如果原项目没有构件导出功能：先给它加一个导出函数，跑一次拿到数据。

**标准字段**（每个构件必须包含）：
- 唯一 ID
- element_type（COL/BMX/BMY/SLAB/FND/TRUSS...）
- floor（所在楼层，基础层=0）
- grid_x / grid_y（柱网坐标）
- position（3D 坐标）
- 尺寸（width/depth/height）
- 旋转（rotation_x/y/z）
- 材质（material_name）
- importance（重要性，越大越晚拆）

### 步骤 3：转换拆除定义

**目标**：将原项目的拆除逻辑转为标准拆除参数文件。

**如果原项目是命令式**（硬编码步骤序列）：做一次「反向工程」——理解每步在做什么，映射到标准格式字段。

**如果原项目是声明式**（策略描述）：直接转换格式即可。

**标准拆除文件字段**：
- 步骤标签（step_label）
- 构件 ID 列表（element_ids）
- 时间帧（frame_start / frame_end）
- 过渡参数（transition_type: flash/fall/explode/settle）
- 效果类型（effect: shake/dust/crack/smoke）

如果原项目的拆除逻辑有独特价值（如蒸汽轮机的西→东渐进式），保留在拆除文件里。格式从 Python 代码变成结构化数据。

### 步骤 4：注册到主项目

1. 几何文件放到标准位置（如 `data/structures/<name>/geometry.json`）
2. 拆除文件放到标准位置（如 `data/structures/<name>/demolition.json`）
3. 如果有新增参数类型，更新 Schema 注册表
4. 执行 Server（如 build_server / animate_server）不需要改——它们按文件名自动发现
5. 更新 CAIAO_PROTOCOL.md 的 Server Registry

### 步骤 5：清理原项目

1. 删除原项目的独立目录（代码和配置）
2. 保留原项目文档/README 中有价值的技术说明（建筑规格、工程背景）
3. 迁移知识文档到主项目的 dev-notes/reference/

### 步骤 6：Simplify 审查

吸收完成后，立即运行 simplify 审查：
- 检查是否有死变量、无效加载、缺少 __init__.py
- 检查是否有可以合并的重复逻辑
- 检查是否所有文件引用路径已更新

## 三、特殊场景处理

### 场景 A：隔离 Python 环境（Abaqus/Blender 内嵌 Python）

**问题**：源项目依赖只能在特定 Python 解释器中运行的库（如 `abaqus` 模块只能在 Abaqus Python 中导入）。

**方案：Wrapper 模式**
1. 创建一个 MCP Server 用系统 Python 运行
2. 该 Server 通过 `subprocess.Popen` spawn 隔离 Python 的子进程
3. 两者通过 stdin/stdout JSON-RPC 通信
4. 系统 Python Server 负责 MCP 协议；隔离 Python 子进程负责实际计算

**关键细节**：
- 使用 `@abaqus_python@` sentinel 值在 caiao.yaml 中声明隔离解释器路径
- 隔离解释器路径从环境配置文件（如 `abaqus_env.json`）动态解析
- 避免在隔离 Python 中安装 mcp SDK

### 场景 B：多个 Server 共享运行时状态

**问题**：原项目的多个 Server 共享同一个运行时对象（如 Abaqus 的 `mdb.models['Model-1']`、Blender 的 `bpy`）。拆分到独立子进程会导致状态不可见。

**方案：合并到单一 Session Server**
1. 将原项目的所有原子 Server 合并为一个 Merged Server
2. 该 Server 维护单一持久子进程
3. 所有工具共享同一个运行时状态

### 场景 C：架构完全不兼容（蒸汽轮机案例）

**问题**：六个维度全部不兼容。

**方案：不尝试逐个修复**。直接执行步骤 2-3——提取数据和拆除定义为标准文件，原代码全部删除。

## 四、从第一天就为吸收做准备

如果你正在开发一个新项目（将来可能被吸收），按以下方式开发：

1. **从第一天就用 _common**：不用 `bpy.ops`，用低级 API
2. **构件创建时立即打元数据**：element_type、floor、importance 等自定义属性
3. **配置文件走主项目 schema**：独有参数放自己的命名空间下
4. **拆除逻辑用声明式**：策略描述而非硬编码步骤
5. **测试时走 CAIAO 路径**：即使开发阶段也通过最小 CAIAO Server 调用
6. **命名空间预留**：任何可能有变体的参数加前缀

## 五、案例对比示例

> 以下为来自参考实现的两个真实案例对比，展示了高难度与低难度吸收的差异。具体的项目名称、技术栈可根据实际场景替换，但兼容性评估框架和吸收方法论是通用的。

| 维度 | 案例 A（高难度） | 案例 B（低难度） |
|------|------------|----------------|
| 架构思想兼容？ | ❌ 无 CAIAO 概念 | ✅ 已有 CAIAO 架构 |
| 工具契约兼容？ | ❌ 自定义协议 | ✅ JSON Schema |
| 吸收耗时 | ~8h 架构讨论 | ~2.5h 总耗时 |
| 核心问题 | 6/6 不兼容，需完全重设计 | 4/6 不兼容但都是格式层 |
| 最终方案 | 持久进程 + 标准文件 | Wrapper 模式 + 合并 Session Server |
| 关键教训 | 不要等「以后吸收」才考虑兼容 | CAIAO 先行开发 = 吸收成本接近零 |

## 六、吸收检查清单

**吸收前**：
- 完成六维度兼容性评估表
- 确定吸收策略（标准文件提取 / Wrapper 模式 / 合并 Session Server）
- 估算耗时

**吸收中**：
- 提取几何定义为标准数据文件（不保留原生成代码）
- 转换拆除/编排定义为标准数据文件
- 创建必要的 Server（environment / session / pipeline）
- 编写 caiao.yaml 清单
- 更新 caiao/discovery.py（如需要新 sentinel）

**吸收后**：
- 删除原项目独立代码
- 迁移知识文档到 dev-notes/reference/
- 更新 CAIAO_PROTOCOL.md Server Registry + Change Log
- 更新 CLAUDE.md
- 运行 simplify 审查
- 运行端到端测试验证
