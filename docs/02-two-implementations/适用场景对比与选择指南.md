# 两种 CAIAO 实现的适用场景对比与选择指南

## 决策矩阵

| 考量维度 | 选轻量实现 | 选 MCP SDK 实现 |
|----------|-----------|----------------|
| **项目阶段** | 原型、早期开发、单人项目 | 生产环境、多人协作、需要监控 |
| **Server 数量** | <10 台 | 10+ 台 |
| **Server 种类** | 主要是编排型和简单计算型 | 含重量级求解器（外部重型求解器、3D 可视化引擎） |
| **稳定性要求** | 开发/演示级别 | 生产级别（崩溃自动重启、健康监控） |
| **AI 客户端接入** | 仅内部使用（CLI、自己的前端） | 需要被 Claude Desktop 等外部 AI 客户端发现 |
| **团队规模** | 1-3 人 | 3+ 人，需要标准化 |
| **Python 环境** | 受限环境（内嵌 Python、无 pip） | 标准 Python 3.11+ venv |
| **调试需求** | 频繁 CLI 调试 | 通过 Gateway 日志和健康端点调试 |
| **并行执行** | 串行为主 | 需要资源自适应的多求解器并行 |
| **运维** | 手动管理 | 通过 manager_server 自动化管理 |

## 选择框架

```
你的项目需要什么？
│
├─ 只是一个独立工具/CLI？
│   └─ 轻量实现。直接拷贝 server.py + hub.py。
│
├─ 需要被外部 AI 客户端（Claude Desktop）调用？
│   └─ MCP SDK 实现。这是 MCP 协议的唯一价值所在。
│
├─ 有多个重量级求解器（FEA、渲染）需要进程隔离？
│   └─ MCP SDK 实现。轻量实现的子进程管理太简陋。
│
├─ 团队需要标准化、自动化管理（健康监控、指标、语义搜索）？
│   └─ MCP SDK 实现 + manager_server。
│
├─ 快速原型、单人开发、需要频繁 CLI 调试？
│   └─ 轻量实现。开发效率是压倒性优势。
│
├─ 在 Abaqus/Blender 内嵌 Python 中运行？
│   └─ 轻量实现。那些环境无法安装 mcp 包。
│
└─ 既有重量级求解器，又需要快速开发迭代？
    └─ 渐进式采用：先用轻量实现开发 → 逐步将计算型 Server 迁移到子进程
       → 最终可切换到 MCP SDK transport（业务逻辑零改动）
```

## 混合策略（推荐）

两种实现可以共存。一个典型的混合策略：

1. **开发阶段**：使用轻量实现的 `CAIAOServer` + `@tool` 装饰器开发所有 Server 的业务逻辑
2. **集成阶段**：编排型 Server（Pipeline、CLI）保留在轻量 Hub 中，进程内直调
3. **部署阶段**：将计算型 Server 包装为 MCP stdio Server，注册到 MCP SDK Hub
4. **生产阶段**：MCP SDK Hub 管理所有子进程生命周期，提供健康监控和指标

关键点：**从轻量实现迁移到 MCP SDK 实现的成本是零**——因为 `list_tools()` 和 `call_tool()` 的接口在两种实现中完全对齐。

## 实际案例

### 案例 A（轻量实现）

- 十余台 Server，全部进程内直调
- 主要使用场景：CLI 一键运行、Web API 服务
- 不需要被外部 AI 客户端发现
- 唯一的外部依赖是 `numpy` 和 `pyyaml`
- 开发效率极高——新 Server 半小时内可完成

**选择理由**：项目是独立工具，不需要 MCP 协议互通。轻量实现的同步模型和 CLI 调试能力对快速迭代至关重要。

### 案例 B（MCP SDK 实现）

- 二十余台 Server，含多个重量级求解器（外部重型求解器、3D 可视化引擎）
- 需要进程隔离（一个求解器崩溃不能影响其他）
- 需要惰性启动（并非所有求解器在每次会话中都会用到）
- 需要被 LLM Agent 动态发现和调用
- 需要健康监控和指标（manager_server）

**选择理由**：项目是多求解器平台，进程隔离和生命周期管理是刚需。MCP SDK 提供的标准化协议和工具发现机制是核心功能。
