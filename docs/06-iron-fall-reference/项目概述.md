# 从非 CAIAO 架构向 CAIAO 迁移：Iron-Fall 案例

> Iron-Fall 是 CAIAO 体系出现之前的一个拆除决策系统。它从无 Server 概念的紧耦合架构，逐步迁移到 CAIAO 体系。本节以它为操作示例，展示 CAIAO 架构相比传统紧耦合架构的差异。
> 完整项目信息见附录：`appendix/Iron-Fall-项目详情.md`。

---

## 一、架构变迁：迁移前后对比

### 迁移前（无 CAIAO 时代）

传统架构的典型特征：求解器硬编码在 API 层，扩展只能改核心代码。

```
FastAPI 端点 → 直接 import 求解器 → 硬编码调用链
                     ↓
             求解器耦合在 API 层，无独立生命周期
```

核心痛点：
- 添加新求解器 = 修改核心代码
- 求解器崩溃影响整个服务
- 不同求解器无法独立部署和扩展
- AI 决策逻辑与领域计算混在一起

### 迁移后（CAIAO 时代）

```
API 端点 → CAIAO Hub → 独立 CAIAO Server 子进程（各求解器完全解耦）
```

核心改进：
- 添加新求解器 = 新增一个 Server 文件 + 注册
- 求解器崩溃不影响其他 Server
- 每个 Server 独立启动、独立升级、独立重启
- AI 决策（编排层）与领域计算（Server 层）彻底分离

| 维度 | 迁移前 | 迁移后 |
|------|--------|--------|
| 架构 | 自定义协议，求解器硬编码 | CAIAO 标准协议，Server 子进程隔离 |
| 通信 | 大量 REST 端点 + 自定义 WebSocket 消息 | WebSocket + 少量 REST，MCP JSON-RPC |
| AI 框架 | 与领域逻辑耦合 | LLM → Hub → Agent Loop 独立层 |
| 扩展性 | 修改核心代码 | 添加一个 Server 文件 |
| Server 概念 | 无 | 每个求解器 = 一个 Server |

## 二、数据模型映射

迁移中的一个关键步骤是将紧耦合的数据模型拆分为各 Server 独立的 Schema：

| 迁移前（集中式） | 迁移后（分散式） |
|:---|:---|
| 单体 Pydantic 数据模型 | 每个 Server 维护自己的 JSON Schema |
| 全局共享的类型定义 | Server 间通过标准化的输入输出来通信 |
| 修改一处影响全局 | 修改只影响单个 Server |

---

## 三、迁移原则

从本次迁移中提炼的通用原则（详细操作指南见 `../10-operations-manual/项目吸收完整操作指南.md`）：

1. **一个求解器 = 一个 Server**：源系统的每个独立计算模块对应一个新 CAIAO Server
2. **保留纯计算逻辑，剥离框架依赖**：去除 HTTP 依赖、Agent 框架 wrapper、特定 ORM
3. **补充 Schema**：为每个 Server 的输入输出定义完整的 JSON Schema
4. **惰性启动**：重量级求解器设为 lazy，按需启动
5. **双轨并行过渡**：迁移后新旧实现可短期并存，做 A/B 对比
6. **利用旧测试**：源系统的测试用例可作为新 Server 测试的参考输入/输出
