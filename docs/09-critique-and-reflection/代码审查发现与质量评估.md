# CAIAO 项目代码审查框架

> 基于对参考实现项目的完整代码审查提炼的通用审查维度与经验教训。
> 具体项目的审查报告见附录：`appendix/代码审查报告-参考实现.md`。

---

## 一、审查维度框架

对任何 CAIAO 项目进行代码审查时，建议覆盖以下维度：

| 维度 | 关注点 |
|------|--------|
| **架构设计** | CAIAO 模式应用是否正确、Hub 解耦是否彻底、Server 边界是否清晰 |
| **领域正确性** | 核心算法的数值精度、规范/标准的实现完备性、边界条件处理 |
| **代码可维护性** | Server 拆分粒度、方法复杂度、命名一致性、Schema 定义的清晰度 |
| **错误处理** | 错误传递方式是否统一、异常是否被正确捕获和返回、降级策略是否到位 |
| **测试覆盖** | 单元测试数量和质量、集成测试是否覆盖 Server 间交互、是否包含精度回归测试 |
| **文档** | README 是否完善、API/Schema 是否有文档、架构决策是否有记录 |
| **安全性** | API 鉴权、CORS 策略、敏感信息保护、速率限制 |
| **前端体验** | 组件是否过大、可视化是否正确、状态管理是否清晰 |
| **日志与可观测性** | 日志是否有级别和结构化格式、是否有调用链追踪、是否有健康检查端点 |

## 二、审查方法论

### 流程

1. **范围确定**：列出所有 Server 及其源文件（Python + 前端），统计代码量
2. **逐维扫描**：按上述维度逐一检查，每个维度给出评分和问题清单
3. **按严重度分级**：P0（影响正确性/安全性）、P1（影响可维护性/可靠性）、P2（影响扩展性）、P3（改善型优化）
4. **输出改进路线图**：分阶段（安全加固 → 可靠性提升 → 规范化 → 扩展），每阶段给出时间估算

### 评分标准

- 9-10：业界最佳实践水平
- 7-8：可用但存在已知短板
- 5-6：存在影响可靠性的问题
- 3-4：存在影响正确性或安全性的问题
- 1-2：不可用于生产

## 三、通用经验教训

以下经验适用于任何 CAIAO 项目，与具体领域无关：

1. **架构优秀不等于领域正确**：CAIAO 模式解决了通信和解耦问题，但不能替代领域算法的正确性验证。漂亮的架构不能掩盖计算错误。

2. **错误处理的一致性比错误处理的方式更重要**：全项目范围内统一错误传递方式（是返回错误对象、抛异常、还是返回空值），比选择哪种方式更关键。三种方式混用是最差的情况。

3. **测试应该验证数值精度**：仅验证"不报错""返回结构含某些 key""值非负"是不够的。需要与解析解、参考软件或 Golden File 做数值对比。

4. **日志是生产就绪的前提**：全部使用 print() 输出的项目不是生产就绪的项目。结构化日志、日志级别、调用链追踪是生产环境的基本要求。
