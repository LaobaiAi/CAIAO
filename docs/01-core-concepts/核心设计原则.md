# CAIAO 核心设计原则

## 四条最高原则

### P1: Server 独立性

每个 CAIAO Server 完全独立。Server 之间不依赖对方的运行时。依赖只能通过导入纯逻辑（函数、类）实现——绝不依赖另一个 Server 的进程是否在运行。

| 允许 | 禁止 |
|------|------|
| 从 `solver_a.server` 导入 `_compute_analysis()` 函数 | 要求 `solver_a` 子进程活着 |
| 如果耦合变紧，复制并自维护逻辑 | 跨 Server 共享可变状态 |
| 创建合并 Server 来导入多个 Server 的逻辑 | 修改已有 Server 来服务新合并 Server |

### P2: Server 作为原子单元

CAIAO Server 是最小可部署单元：
- 运行在自己的 stdio 子进程中
- 通过 `list_tools()` / `call_tool()` 暴露工具
- 可以热启动（gateway 初始化时）或惰性启动（首次工具调用时）
- 一个崩溃绝不级联——Hub 重启或报告失败

### P3: 合并而非修改

当某个 Server 序列被反复高频使用时，通过导入它们的逻辑将它们合并为一个新的独立 Server。绝不修改源 Server 来服务合并。

正确的做法：`pipeline_server_a` 从 `generator_server.core` 和 `solver_a.server` 导入逻辑 → 新子进程，独立运行。

错误的做法：修改 `solver_a` 来包含生成逻辑 → 破坏 `solver_a` 的原子性。

### P4: 按 ROI 提取共享逻辑

如果多个 Server 共享通用逻辑：
1. 第一次出现：内联
2. 第二次出现：仍然内联
3. 第三次出现：评估提取到共享模块的 ROI
4. 只有共享逻辑非平凡且稳定时才提取

不要「以防万一」式地创建共享抽象。

## 合并 vs 依赖

合并 Server 在导入时从源 Server 导入纯逻辑。这是编译时/依赖耦合，不是运行时耦合：

```
pipeline_server_a
  ├── 从 generator_server.core 导入 CoreGenerator  ✅ 纯类
  ├── 从 solver_a.server 导入 _compute_analysis     ✅ 纯函数
  └── 在自己的子进程中运行                          ✅ 独立
```

如果 `generator_server.core` 有 breaking change，`pipeline_server_a` 需要更新——但它不会因为 `generator_server` 崩溃而在运行时出错。这就是代码依赖（可接受，通过 import 管理）和运行时依赖（禁止）的区别。

## Server 设计原则

| 原则 | 说明 |
|------|------|
| **原子性** | 一个 Server 只做一件事，单文件 `.py` |
| **零互联** | Server 间绝不直接 import，通过 Hub 路由 JSON |
| **无状态** | 输入足够，不依赖外部隐式状态 |
| **可独立验证** | 给定输入，输出可单独测试 |
| **契约驱动** | `list_tools()` + `call_tool()` 统一接口 |
| **AI 原生** | 工具描述和 Schema 完整，LLM Agent 可直接调用 |
| **容错** | `call_tool` 捕获异常返回错误字典，不抛出未处理异常 |
| **进程隔离** | 计算型 Server 子进程运行，崩溃不牵连主进程 |
