# CAIAO 设计演进与关键决策

> 从多个参考实现的开发历程中提炼的 CAIAO 体系关键设计决策和演进逻辑。
> 具体项目时间线和 Server 数量统计见附录：`appendix/项目时间线.md`

---

## 一、演进逻辑总览

CAIAO 体系不是一次性设计出来的，而是经历了"具体项目 → 抽象模式 → 新项目复用 → 模式固化"的多个循环。理解这个演进过程有助于理解为什么 CAIAO 是现在的样子。

核心演化路径：

```
紧耦合单体（无 CAIAO）
  → 协议建立（一切皆 Server，MCP SDK 为底层传输）
    → 原子化（Server = 最小单元，Merge 替代修改）
      → 双重实现（SDK 实现 + 轻量实现，选择有据）
        → 垂直爆发（多领域 Server 并行创建）
          → 新种类诞生（infrastructure Server 管理共享运行时）
            → 新模式提炼（常驻服务 Daemon 替代一次性调用）
```

---

## 二、关键设计决策

### 决策 1：选择 MCP SDK 作为底层传输

**问题**：Server 之间如何通信？自研协议还是采用现有标准？

**决策**：选择 MCP SDK（Model Context Protocol）的 stdio transport 层作为底层，在其上建立 CAIAO 命名层和约定层。

**理由**：
- MCP SDK 已经被广泛测试，处理了 JSON-RPC 解析、连接管理、错误恢复
- 自研协议需要长期维护和 debug
- CAIAO 的价值在上层约定和模式，不在底层协议

**影响**：MCP SDK 的任何 breaking change 会影响 CAIAO。但这个风险被 SDK 的成熟度抵消。

### 决策 2：Server = 最小原子单元

**问题**：Server 的粒度应该多大？一个 Server 管多少功能？

**决策**：每个 Server 是"可独立运行、可独立测试、可独立部署"的最小单元。类比 LLM 的 token——越小越可组合。

**理由**：
- 粒度越细，组合灵活性越高
- 一个 Server 崩溃不影响其他
- 便于独立测试和独立升级

**边界规则**：
- 单一职责：一个 Server 只做一件事
- 零间接依赖：不依赖其他 Server 的运行时
- 无状态：每次调用自包含
- 纯计算：输入 → 计算 → 输出

### 决策 3：Merge, Don't Modify

**问题**：多个 Server 经常被一起调用，如何优化？

**决策**：当一组 Server 的调用序列高频出现时，创建一个新的"合并 Server"（merge server），导入源 Server 的纯逻辑，编排为一次调用。**绝不修改源 Server。**

**理由**：
- 源 Server 的原子接口不变，其他组合仍可用
- 合并 Server 是一个独立的 Server，有独立的生命周期
- 类比 BPE 算法——高频 token pair → 新 token

### 决策 4：两种实现路径共存

**问题**：CAIAO 只能基于 MCP SDK 吗？

**决策**：提供两条路径——基于 MCP SDK 的完整实现，和基于自建轻量框架的极简实现。两者共享相同的核心概念和契约。

**理由**：
- 不是所有项目都需要 MCP SDK 的完整能力
- 轻量实现零外部依赖，适合嵌入、演示、快速原型
- 两条路径可以互转（升级只需替换通信层，约 10 行代码变更）

### 决策 5：Infrastructure Server 种类

**问题**：像 Blender、Abaqus、MATLAB 这类有状态的外部工具，如何纳入 CAIAO 模型？

**决策**：引入第三种 Server 种类——infrastructure。它管理共享运行时环境（如一个常驻的外部工具进程），提供环境发现和健康检查，其他功能 Server 通过它来操作外部工具。

**理由**：
- 将"环境"显式建模为 Server，遵循"一切皆 Server"原则
- 避免各功能 Server 各自启动外部工具（冷启动开销 × N）
- 环境 Server 的崩溃不影响功能 Server（可独立重启）

### 决策 6：Daemon 常驻服务模式

**问题**：外部重型工具被频繁调用时，每次冷启动的累积开销（数秒到数十秒）严重影响体验。

**决策**：将"一次性工具"升级为"常驻服务"——用一台 infrastructure Server 维持外部工具进程常驻，暴露细粒度的原子操作，功能 Server 只做编排和转发。

**理由**：
- 消除冷启动开销（从 N 次冷启动 → 1 次）
- 功能 Server 不再包含工具启动逻辑，代码量大幅下降
- 标准文件格式（非代码）定义场景参数，新增场景无需改代码

## 三、模式涌现逻辑

回顾整个演进，很多模式不是预先设计的，而是"事后发现"的：

1. **轻量实现** 本是 MCP SDK 版本的原型验证——后来发现它本身就是一个有独立价值的架构选择
2. **Infrastructure Server** 本是为 Blender 环境管理而设计的——后来发现它适用于任何有状态的共享运行时（Abaqus、MATLAB、ANSYS）
3. **Daemon 模式** 本是解决 Blender 冷启动问题的——后来提炼为通用的"常驻服务"模式
4. **合并路线图** 本是人工观察高频调用模式——后来由 manager_server 的 detect_merge_opportunities 工具自动发现

这揭示了一条规律：**CAIAO 体系的进化方式是从具体项目的具体问题出发，提炼出通用模式，再被新项目复用验证。**
