# CAIAO Server 合并路线图

## 合并概念

在 CAIAOServerizer 范式中，Server 是原子单元。当某个 Server 序列反复高频出现时，将其合并为一个新的原子 Server——类似 BPE 合并高频 token 对。

合并前：N 次子进程调用（N 个「token」）
合并后：1 次子进程调用（1 个「token」）

## 合并原则

1. 导入逻辑，不依赖源 Server 进程——合并 Server 从已有模块导入函数/类
2. 不修改源 Server——原子 Server 保持独立
3. 返回统一结果——一次响应返回前端全部需要的数据
4. 保留旧 Server——原子 Server 仍可供需要单独使用的用户

## 合并检测

`manager_server` 的 `detect_merge_opportunities` 工具自动分析工具调用模式，建议合并候选：

1. 分析调用日志中频繁出现的工具序列
2. 评估合并的延迟收益（减少子进程 hop）
3. 评估合并的耦合风险（是否增加维护负担）
4. 输出排序后的合并建议

## 合并模板

合并 Server 的标准实现结构：

1. 在 `servers/` 下创建新目录
2. 编写 `server.py`——从源模块导入纯函数，编排调用顺序
3. 编写 `caiao.yaml`——声明 kind 为 `atomic-mcp` 或 `merged`（如果是合并）
4. 不用写源代码——所有逻辑来自导入
5. 包含单元测试

具体代码模板见 `servers/quick_analysis_server/server.py`（Pipeline A 的完整实现）。
