# CAIAO 可复用代码模式

> 从三个项目的实际代码中提炼的、可跨项目复用的设计模式。
> 每个模式包含：问题 → 解决方案 → 代码位置 → 何时复用。

---

## 模式 1：共享默认值单源（Single Source of Truth for Defaults）

### 问题
多个 Server 和 CLI 入口各自硬编码默认参数（截面库、材料属性、设计参数），导致不一致和同步困难。

### 解决方案
创建一个 `defaults.py` 作为项目中所有默认值的唯一来源。所有 Server 和 CLI 入口从此模块导入。

### 实现位置
在轻量实现的 Server 目录中，作为 `defaults.py` 独立文件。

### 设计要点
- 不硬编码默认值到任何 Server 的代码中
- 前端和后端共享颜色映射定义（避免视觉不一致）
- Quick 演示参数从默认值派生（只覆盖差异部分）
- 新增参数只需修改这一个文件

### 何时复用
任何有多个 Server 共享配置数据的 CAIAO 项目。

## 模式 2：LLM 三层架构

### 问题
LLM 集成通常是单体式的——一个 Server 既处理 HTTP 通信，又解析响应，又编排工作流。这导致：
- 更换 LLM 提供商需要修改多处
- 单元测试困难（必须 mock 网络）
- API Key 管理混乱

### 解决方案
将 LLM 集成拆分为明确的三层：

```
编排层（合并 Server）→ ReAct 循环，纯编排
计算层（原子 Server）→ JSON 解析、校验、默认值填充，零网络
通信层（原子 Server）→ 唯一的 HTTP 调用者
```

### 实现位置
- 通信层：在轻量实现的 Server 目录中（LLM Gateway Server）
- 计算层：在轻量实现的 Server 目录中（参数提取 Server）
- 编排层：在轻量实现的 Server 目录中（Agent 编排 Server）

### 设计要点
- 通信层是唯一需要 `requests` 库和 API Key 的组件
- 计算层通过 Hub 调用通信层（`hub.call_tool("chat_completion", ...)`），不直接发 HTTP
- 编排层通过 Hub 动态发现工具，不在 System Prompt 中硬编码工具列表
- API Key 优先级：环境变量 > 入参 > 默认值
- 智能摘要：工具结果摘要后回传 LLM（而非完整 JSON），节省 80-90% Token

### 何时复用
任何需要 LLM 集成的 CAIAO 项目。

## 模式 3：智能结果摘要

### 问题
LLM Agent 在执行工具调用后，将完整的 JSON 结果回传给 LLM 会消耗大量 Token（如完整的结构分析结果可能有数万字符）。但 LLM 只需要知道「分析完成了，最大位移是 X」这样的关键信息来做下一步决策。

### 解决方案
在编排 Server 中为每种工具类型实现一个轻量的结果摘要器。每个摘要器提取 1-3 个关键指标，用一句话返回。

### 实现位置
Agent 编排 Server 的 `_summarize_result()` 方法

### 摘要示例
- generate_frame → "模型生成完成: 126 节点, 85 单元"
- run_analysis → "分析完成: 最大位移 0.003421m"
- check_code → "校核完成: 80/85 通过, 最大应力比 0.82"

### 设计要点
- 每种工具类型有专门定制的摘要逻辑
- 对于未知工具类型，自动提取前几个数值字段作为通用摘要
- 摘要器本身是纯函数，易于测试

### 何时复用
任何使用 LLM Agent 模式编排工具的项目。

## 模式 4：Web API Server 的「协议转换器」角色

### 问题
CAIAO Server 通过 `call_tool(name, input)` 通信（JSON in/out），但前端期望 REST/SSE 端点。如何桥接这两个世界？

### 解决方案
创建一个专门的 Web API Server——它本身是一个 CAIAO 合并 Server，不包含任何领域逻辑，唯一的职责是：
- REST 请求 → `hub.call_tool()` → JSON 响应
- SSE 流 → `hub.call_tool("stream_chat")` → `StreamingResponse`

### 实现位置
在轻量实现的 Server 目录中（Web API Server）。

### 设计要点
- Server 自身通过 `@tool` 装饰器暴露管线运行、参数提取、Agent 编排等工具
- FastAPI 路由是对这些工具的薄包装：解析 HTTP 参数 → 调用工具 → 构建 HTTP 响应
- 启动时手动注册需要 hub 引用的 Server（覆盖自动发现的无 hub 实例）
- 子进程计算型 Server 通过 `hub.register_subprocess()` 惰性注册

### 何时复用
任何需要将 CAIAO Hub 能力暴露为 Web API 的项目。

## 模式 5：自动项目归档（ML-Ready 数据保存）

### 问题
每次管线运行产生的结果散落在临时文件中，没有统一的持久化机制。对于需要积累训练数据的场景（如 ML 模型优化截面选型），这是巨大的浪费。

### 解决方案
在 Web API Server 中实现自动归档——每次管线成功运行后，自动将输入参数 + 输出结果保存为结构化 JSON。

### 实现位置
Web API Server 的 `_auto_save_project()` 函数

### 数据结构
三层结构，ML 友好：
- metadata：项目名、时间戳、版本
- input：几何参数、截面材料、荷载（天然特征向量）
- output：校核摘要与构件明细（天然监督标签）

### 设计要点
- 文件名含时间戳，按项目名分组
- 路径安全检查（防止目录穿越攻击）
- 支持手动保存和批量加载

### 何时复用
任何需要积累运行数据的 CAIAO 项目。

## 模式 6：计算过程可追踪

### 问题
规范校核等复杂计算的输出是「通过/不通过」的二元结果，用户（和 LLM）无法理解计算过程。这降低了可信度和可调试性。

### 解决方案
在工具输出中附加 `calc_processes` 数组，每步包含：步骤编号、名称、公式、输入值、结果、单位。

### 实现位置
在轻量实现的 Server 目录中（规范校核 Server）

### 设计要点
- 每步计算都是独立记录
- 包含原始参数值，可复现
- 前端可直接渲染为展开式详情面板

### 何时复用
任何有复杂多步计算的领域 Server（结构分析、流体计算、信号处理等）。

## 模式 7：Hub 感知 vs 无 Hub 的双模 Server

### 问题
有些 Server 需要调用其他 Server 的工具（如参数提取器需要调用 LLM Gateway），有些完全独立。如何优雅处理这两种情况？

### 解决方案
Server 构造时接受可选的 hub 引用：
- 有 hub 引用 → Hub 感知模式，通过 `hub.call_tool()` 调用其他工具
- 无 hub 引用 → 独立模式，或回退到直接实例化依赖 Server

### 实现位置
在轻量实现的 Server 目录中（参数提取 Server、管线编排 Server）

### 设计要点
- 构造参数 `hub=None` 是默认值
- `self._call()` 方法封装调用逻辑：Hub 优先 → 直接调用回退
- Hub 感知 Server 通过 `hub.register(instance)` 手动注册（覆盖自动发现的无 hub 实例）

### 何时复用
任何有 Server 需要调用其他 Server 工具的 CAIAO 项目。

## 模式 8：子进程的优雅降级

### 问题
计算型 Server 需要独立子进程运行（实现进程隔离），但其依赖的虚拟环境（如 `.venv_opensees`）可能不存在。

### 解决方案
在注册子进程时检测虚拟环境是否存在，不存在时自动回退到当前 Python 解释器。

### 实现位置
Web API Server 的 `startup()` 函数

### 设计要点
- 先检查特定 venv 路径是否存在
- 不存在时回退到 `sys.executable`
- 输出清晰的日志说明使用了哪个 Python

### 何时复用
任何需要子进程隔离但依赖环境可能缺失的 CAIAO 项目。
